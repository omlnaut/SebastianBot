Architecture Review: Deepening Opportunities

Focus: turn shallow Module slices into deep Module slices for testability and AI-navigability.

Candidate 1 — Deepen the Delivery Notification / Return Notification mail-processing Module

Strong

Files

  • sebastian/usecases/features/delivery_ready/handler.py
  • sebastian/usecases/features/return_tracker/handler.py
  • sebastian/usecases/features/delivery_ready/parsing.py
  • sebastian/usecases/features/return_tracker/parsing.py
  • tests/unit_tests/usecases/test_delivery_ready_retry.py
  • tests/unit_tests/usecases/test_return_tracker_retry.py

Problem

Both UseCase handlers implement nearly identical retry horizon checks, transient retry, terminal failure Side Effect emission, and mail age logic. The Interface for each Module is broad (caller must know retry config, mail age policy, parsing failure policy) while the Implementation remains duplicated. This is shallow Depth and low Locality: a bug fix in Delivery Notification flow must be copied into Return Data flow.

Deletion test: deleting one handler’s policy helpers does not remove complexity; it reappears in the sibling handler almost line-for-line.

Solution

Create one deep Module for “mail-to-domain extraction with retry horizon policy.” Keep Delivery Notification parsing and Return Data parsing as separate Adapter choices plugged at a Seam in that Module. The Implementation owns timing, retry, terminal escalation, and mark-as-read policy once.

Benefits

Higher Leverage: each caller only supplies domain-specific extraction rules and mapping to Task Creation. Higher Locality: retry and failure behaviour lives in one place. Tests improve because the Interface is the test surface: one shared policy test matrix covers both Delivery Notification and Return Data paths with fewer duplicated fake clients.

Candidate 2 — Deepen the trigger/orchestration Module around UseCase execution and Side Effect dispatch

Worth exploring

Files

  • cloud/functions/features/*.py
  • cloud/functions/side_effects/*/function.py
  • cloud/functions/side_effects/shared.py
  • cloud/dependencies/usecases.py
  • function_app.py

Problem

UseCase wiring is spread across many tiny Modules: trigger function files, resolver functions, SIDE_EFFECT_MAP, and per-event request builders. The Interface knowledge is fragmented (where to add a new UseCase, where to map Side Effect, where to add trigger). Each file is shallow and mostly pass-through Implementation.

Deletion test: deleting a single wrapper Module rarely removes complexity; callers must recreate almost identical glue in another file.

Solution

Concentrate orchestration in one deeper Module that owns registration of triggers, request translation, handler resolution, and Side Effect fan-out. Keep concrete trigger sources (Timer, EventGrid) as Adapter Modules attached to a single Seam.

Benefits

Leverage improves for maintainers and AI navigation: adding a new UseCase follows one path instead of hunting several shallow files. Locality improves because failure handling and dispatch behaviour are centralized. Tests can target one Interface contract for orchestration rather than file-by-file smoke checks.

Multiple shallow Modules split one concept: orchestration.

Candidate 3 — Deepen the Task note contract Module for Tracking Number handling

Strong

Files

  • sebastian/usecases/features/delivery_ready/handler.py
  • sebastian/usecases/features/check_parcel_received/handler.py
  • sebastian/domain/task.py
  • tests/unit_tests/usecases/ (no direct tests for this cross-Module contract)

Problem

Tracking Number contract is implicit string formatting: Delivery Notification Task Creation writes free-text notes, then CheckParcelReceived parses with regex. Interface invariants (line prefix, uppercase format, Task Tag placement) are not explicit. This Seam leaks implementation details across UseCase Modules and produces fragile tests.

Deletion test: remove the regex parser and every caller would need to re-encode/decode the same note grammar manually.

Solution

Introduce one deep Module that owns the Delivery-ready Task note grammar: encode from Pickup Data and decode to Tracking Number + Task Tag facts. Delivery Notification and CheckParcelReceived become Adapter Modules using this Seam rather than hand-rolled strings.

Benefits

Leverage: callers get validated note construction and parsing without knowing line format rules. Locality: grammar changes happen once. Tests improve via round-trip property tests at the Interface test surface, plus focused scenario tests for malformed notes.

Top recommendation

Start with Candidate 1. It has the best leverage/locality ratio right now: duplicated Delivery Notification and Return Notification policy logic already diverges easily, and this deepening creates immediate shared tests without changing domain behaviour.

Candidate 3 is the next best follow-up because it hardens the Tracking Number seam that links Task Creation and Task Completion flows.
